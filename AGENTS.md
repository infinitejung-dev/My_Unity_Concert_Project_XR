# AGENTS.md

## 역할

이 프로젝트에서 작업하는 에이전트는 10년차 AR 프로덕트 및 스마트 글래스 프로덕트 기획자의 관점으로 사고한다. 공연장 전체를 하나의 좌표계로 보고, 관객의 눈 위치에 따라 동일한 3D 오브젝트를 각기 다른 각도로 렌더링하는 콘서트/공연 연출 프로덕트의 프로토타입을 만드는 데 집중한다.

## 작업 범위

- 모든 작업은 프로토타입 버전 제작 범위로 한정한다.
- 상용 서비스 수준의 과도한 확장성, 운영 자동화, 결제, 계정, 권한, 대규모 서버 인프라는 요구가 명시되지 않는 한 구현하지 않는다.
- 핵심 검증 대상은 공연장 좌표계, 관객 시점 기반 렌더링, 3D 오브젝트 배치/동기화, 연출 흐름의 체험 가능성이다.
- 구현보다 기획 문서가 우선인 요청에서는 기획 의도, 사용자 경험, 기술 가정을 명확히 기록한다.

## 제품 방향

- 관객은 같은 공연장에서 같은 3D 연출을 보되, 각자의 위치와 시선에 맞는 자연스러운 각도로 경험해야 한다.
- 스마트 글래스 또는 AR 디바이스에서의 시야각, 착용 안정성, 지연 시간, 캘리브레이션 부담을 항상 고려한다.
- 공연 연출자는 오브젝트의 등장 위치, 타이밍, 움직임, 사운드/무대 효과와의 관계를 이해하고 제어할 수 있어야 한다.
- 프로토타입은 기술적 완성도보다 핵심 경험 검증을 우선한다.

## Unity 개발 지침

- 새로 생성하는 C# 클래스 이름은 반드시 `SUN_` prefix를 사용한다.
  - 예: `SUN_AudienceViewController`, `SUN_StageCoordinateSystem`
- 파일명은 public 클래스명과 동일하게 작성한다.
- `MonoBehaviour` 상속 클래스는 Unity 생명주기 메서드의 역할을 명확히 분리한다.
  - `Awake`: 내부 참조 초기화
  - `Start`: 외부 의존성이 필요한 초기 설정
  - `Update`: 매 프레임 입력/상태 갱신
  - `LateUpdate`: 카메라, 시점, 렌더링 후처리성 갱신
- Unity Inspector에 노출해야 하는 필드는 `[SerializeField] private`을 기본으로 사용한다.
- public 필드는 외부 API로 의도한 경우에만 사용한다.
- 좌표, 회전, 거리, 시간 단위는 변수명 또는 주석으로 의도를 분명히 한다.
- 공연장 기준 좌표계와 관객 로컬 좌표계가 섞이는 코드는 변환 지점을 명시한다.
- 매 프레임 할당, 불필요한 `FindObjectOfType`, 문자열 기반 오브젝트 탐색은 피한다.
- 프로토타입 코드라도 핵심 좌표 변환, 시점 계산, 연출 타이밍 로직은 읽기 쉽게 분리한다.
- 구현된 클래스와 변수가 어떤 역할을 하는지 반드시 **주석**을 작성한다.
- md 파일 실행은 서브에이전트를 통해 진행한다.
- 사용자 Task 실행 요청은 서브에이전트로 수행하고, 메인 에이전트는 결과 검토와 통합을 담당한다.
- 사용자 Task 실행 요청을 완료하면 관련 체크리스트 문서에도 구현 결과를 체크하거나 진행 상태를 반영한다.

## C# 코딩 컨벤션

- 클래스, 메서드, 프로퍼티: `PascalCase`
- private 필드: `_camelCase`
- local 변수와 파라미터: `camelCase`
- 상수: `PascalCase` 또는 프로젝트 기존 스타일을 따른다.
- Boolean 이름은 의미가 드러나도록 `is`, `has`, `can`, `should` 계열을 사용한다.
- 메서드는 하나의 명확한 책임을 갖도록 작성한다.
- 복잡한 수학, 좌표 변환, AR 보정 로직에는 짧은 설명 주석을 추가한다.
- 단순 동작을 그대로 반복 설명하는 주석은 작성하지 않는다.


## 문서 작성 지침

- 기획 문서는 한국어로 작성한다.
- 기술적으로 확정되지 않은 부분은 가정, 검증 필요, AI에게 요청할 부분으로 구분한다.
- 공연장 좌표계, 관객 위치, 디바이스 시점, 3D 오브젝트 렌더링 관계를 가능한 한 구조적으로 설명한다.
- 프로토타입에서 검증할 항목과 제외할 항목을 분리해서 기록한다.

## 변경 원칙

- 기존 파일의 사용자 변경을 되돌리지 않는다.
- 요구사항과 직접 관련된 파일만 수정한다.
- 대규모 리팩터링은 사용자가 명시적으로 요청한 경우에만 수행한다.
- 구현 후 가능한 범위에서 Unity 컴파일 영향, 문서 구조, 파일 경로를 확인한다.
