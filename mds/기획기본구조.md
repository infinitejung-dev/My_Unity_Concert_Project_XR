# 기획서: [콘서트 및 공연 AR 연출 프로덕트]

## 프로젝트명
[콘서트 및 공연 AR 연출 프로덕트]

## 프로덕트 개요
공연장 전체를 하나의 공간 좌표계로 구성하고, 각 관객의 눈 위치를 기준으로 동일한 3D 오브젝트를 서로 다른 시점과 각도로 렌더링하는 AR 콘서트 및 공연 연출 프로덕트.

## 핵심 컨셉

### 한 줄 정의
공연장 전체를 하나의 좌표계로 만들고, 각 관객의 눈 위치에서 같은 3D 오브젝트를 서로 다른 시점으로 보이게 하는 공동 AR 연출 프로토타입.

### 공간 좌표계
- 공연장 또는 특정 체험 공간을 하나의 기준 좌표계로 정의한다.
- 무대, 객석, 중앙 마커, 3D 오브젝트 배치 위치는 모두 같은 공간 좌표계 안에서 관리한다.
- 공간 중앙의 마커는 관객 디바이스가 공통 기준점을 맞추기 위한 앵커 역할을 한다.

### 관객 좌표와 시점
- 각 관객은 공간 좌표계 안에서 고유한 위치 좌표를 가진다.
- 렌더링 기준은 관객의 몸 위치가 아니라 눈 위치로 둔다.
- 디바이스는 관객의 눈 위치와 바라보는 방향을 기준으로 3D 오브젝트의 화면상 위치, 크기, 회전감을 계산한다.

### 동일 오브젝트의 개인별 렌더링
- 모든 관객은 같은 3D 오브젝트와 같은 연출 이벤트를 경험한다.
- 관객 위치가 다르기 때문에 각 디바이스에는 오브젝트가 서로 다른 각도와 원근감으로 보인다.
- 목표는 오브젝트가 화면 위에 붙어 있는 이미지가 아니라 실제 공연장 공간에 떠 있는 대상처럼 느껴지게 하는 것이다.

### 중앙 마커 기반 정렬
- 체험 시작 시 모든 관객이 공간 중앙의 마커를 바라본다.
- 디바이스는 마커를 기준으로 자신의 위치와 방향을 보정한다.
- 보정 후 3D 오브젝트는 마커 주변 또는 공연장 내 지정 좌표에 배치된다.

### 연출 동기화
- 연출 이벤트가 발생하면 모든 디바이스는 같은 오브젝트, 같은 타이밍, 같은 상태값을 사용한다.
- 각 디바이스는 공통 상태값을 자신의 관객 시점으로 변환해 렌더링한다.
- 동기화 대상은 오브젝트 등장, 이동, 회전, 사라짐, 무대 효과와의 타이밍이다.

### 프로토타입 검증 포인트
- 하나의 공연장 좌표계 안에 관객과 3D 오브젝트를 함께 배치할 수 있는가.
- 관객 눈 위치가 달라질 때 같은 오브젝트가 자연스럽게 다른 각도로 보이는가.
- 중앙 마커를 기준으로 여러 디바이스의 공간 정렬이 가능한가.
- 같은 이벤트를 여러 디바이스에서 동시에 체험하는 느낌을 줄 수 있는가.


## 핵심 메커니즘

### 작성 기준
- 목적은 공연장 좌표계, 관객 눈 위치 기반 시점, 동일 3D 오브젝트의 개인별 렌더링, 이벤트 동기화를 기본 프로토타입 수준에서 검증하는 것이다.
- 구현 범위는 Unity 단일 씬 또는 소수 디바이스 시뮬레이션 기준으로 제한한다.
- 좌표 단위는 미터(m)를 기준으로 하며, 높이는 Unity 기준 Y축으로 관리한다.
- 대규모 서버 동기화, 계정, 권한, 결제, 운영 자동화는 이번 메커니즘 검증 범위에서 제외한다.

### 1. 공간

#### 공간과 공간 안의 인물들을 하나의 좌표계로 제작
- 프로토타입 검증 유형: 공연장 기준 좌표계 구성 검증.
- 공연장 전체를 하나의 `Stage Space`로 정의하고, 무대, 객석, 관객, 중앙 마커, 3D 오브젝트를 모두 같은 좌표계 안에 배치한다.
- 중앙 마커 위치를 원점 `(0, 0, 0)`으로 두고, 마커에서 무대 중심을 바라보는 방향을 기준 전방축으로 사용한다.
- 관객은 좌석 또는 스탠딩 위치를 기준 좌표로 가지며, 실제 렌더링 기준은 관객의 몸 위치가 아니라 눈 위치로 계산한다.
- 관객 눈 위치는 `관객 기준 위치 + 눈 높이 오프셋`으로 단순화한다.

개발 체크리스트:
- [ ] 공연장 원점, 축 방향, 단위(m)를 Unity 씬과 문서에 동일하게 정의한다.
- [ ] 무대 중심, 객석 기준점, 중앙 마커, 테스트 관객 위치를 `Stage Space` 좌표로 배치한다.
- [ ] 관객별 눈 높이 오프셋을 적용해 카메라 기준 위치를 계산한다.
- [ ] 최소 2개 이상의 관객 위치에서 같은 3D 오브젝트를 바라보는 테스트 씬을 만든다.

검증 기준:
- 서로 다른 관객 좌표를 입력해도 같은 오브젝트가 하나의 공연장 공간 안에 고정된 대상으로 유지되어야 한다.
- 관객 위치를 바꾸면 오브젝트의 화면상 위치와 원근감이 자연스럽게 달라져야 한다.

#### 중앙 마커
- 프로토타입 검증 유형: 공통 기준점 캘리브레이션 검증.
- 중앙 마커는 모든 관객 디바이스가 같은 공연장 좌표계로 정렬하기 위한 기준 앵커다.
- 체험 시작 시 관객 디바이스는 중앙 마커를 인식하고, 인식된 마커의 위치와 회전을 기준으로 디바이스 로컬 좌표계를 `Stage Space`에 맞춘다.
- 마커 인식이 실패하거나 시점이 틀어진 경우, 프로토타입에서는 재스캔 또는 수동 리셋 버튼으로 보정한다.

개발 체크리스트:
- [ ] 중앙 마커 오브젝트를 씬 원점에 배치하고 방향 기준을 표시한다.
- [ ] 마커 인식 후 디바이스 로컬 좌표를 `Stage Space` 좌표로 변환하는 흐름을 정의한다.
- [ ] 마커 재인식 또는 리셋 시 관객 카메라와 3D 오브젝트 기준이 함께 갱신되도록 한다.
- [ ] 마커 정렬 전 상태와 정렬 후 상태를 개발자가 확인할 수 있는 디버그 표시를 둔다.

검증 기준:
- 여러 관객 시점이 같은 중앙 마커를 기준으로 정렬되어야 한다.
- 마커 재정렬 후에도 3D 오브젝트의 공연장 기준 위치가 유지되어야 한다.

### 2. 관객

#### 카메라 시점
- 프로토타입 검증 유형: 관객 눈 위치 기반 렌더링 검증.
- 관객 카메라는 스마트 글래스 또는 AR 디바이스의 눈 위치를 대표하는 시점으로 사용한다.
- 카메라 위치는 `Stage Space`의 관객 눈 위치를 기준으로 하며, 카메라 회전은 관객이 바라보는 방향을 기준으로 갱신한다.
- 실제 착용 안정성이나 정밀한 양안 시차는 이번 단계에서 단순화하고, 한 개의 대표 카메라로 공간감과 각도 차이를 먼저 검증한다.

개발 체크리스트:
- [ ] 관객별 카메라 위치를 눈 높이 기준으로 배치한다.
- [ ] 카메라 회전값을 관객 시선 방향으로 갱신하는 입력 또는 시뮬레이션 값을 준비한다.
- [ ] 관객 위치가 달라질 때 오브젝트의 각도, 거리감, 화면 위치가 다르게 보이는지 확인한다.
- [ ] 스마트 글래스 시야각을 가정한 카메라 FOV 값을 설정하고, 오브젝트가 시야 밖으로 나가는 상황을 확인한다.

검증 기준:
- 같은 오브젝트가 관객 A와 관객 B에게 서로 다른 각도와 거리감으로 보여야 한다.
- 카메라가 회전해도 오브젝트가 화면에 붙은 이미지처럼 움직이지 않고, 공연장 공간에 놓인 대상으로 느껴져야 한다.

#### 이벤트 동기화
- 프로토타입 검증 유형: 공통 타임라인 재생 검증.
- 연출 이벤트는 `이벤트 ID`, `오브젝트 ID`, `시작 시간`, `지속 시간`, `위치/회전/크기 변화`, `표시 상태`를 가진다.
- 프로토타입에서는 연출자용 시작 버튼 또는 공통 타임라인 시작 시각을 기준으로 모든 관객 디바이스가 같은 이벤트 상태값을 사용한다.
- 각 디바이스는 같은 이벤트 상태값을 자신의 관객 카메라 시점으로 변환해 렌더링한다.
- 네트워크 지연 보정은 상용 수준으로 구현하지 않고, 시작 타이밍과 상태값 공유가 체험상 동시에 느껴지는지 확인하는 수준으로 제한한다.

개발 체크리스트:
- [ ] 최소 1개의 테스트 이벤트를 정의한다. 예: 오브젝트 등장, 중앙 상승, 회전, 사라짐.
- [ ] 이벤트 상태값을 관객별 렌더링과 분리해 공통 데이터로 관리한다.
- [ ] 모든 관객 카메라가 같은 이벤트 ID와 같은 진행 시간을 참조하도록 한다.
- [ ] 이벤트 시작, 일시정지, 리셋을 수동으로 실행할 수 있는 테스트 제어를 만든다.

검증 기준:
- 관객 위치가 달라도 같은 순간에 같은 오브젝트 상태를 경험해야 한다.
- 이벤트 진행 중 관객 시점만 달라지고 오브젝트의 공연장 기준 위치와 상태는 동일해야 한다.

### 3. 3D 오브젝트

#### 3D 오브젝트 렌더링
- 프로토타입 검증 유형: 동일 오브젝트의 관객별 원근 렌더링 검증.
- 3D 오브젝트는 `Stage Space` 안의 하나의 실제 위치와 회전값을 가진 대상으로 배치한다.
- 각 관객 디바이스는 중앙 마커로 보정된 좌표 변환값을 사용해 오브젝트의 공연장 기준 좌표를 자신의 로컬 AR 좌표로 변환한다.
- 오브젝트는 기본적으로 카메라를 향해 항상 붙는 빌보드가 아니라, 공간 안에 고정된 3D 대상으로 렌더링한다.
- 프로토타입에서는 기본 메시, 머티리얼, 그림자 또는 바닥 기준 시각 단서를 사용해 오브젝트가 공연장 공간에 떠 있거나 놓여 있다는 느낌을 검증한다.

개발 체크리스트:
- [ ] 테스트용 3D 오브젝트 1개를 `Stage Space` 좌표로 배치한다.
- [ ] 오브젝트 위치, 회전, 크기를 이벤트 상태값으로 갱신할 수 있게 한다.
- [ ] 관객별 카메라에서 같은 오브젝트가 서로 다른 원근과 각도로 보이는지 확인한다.
- [ ] 오브젝트가 관객 시야 밖에 있거나 너무 가까운 경우를 테스트한다.
- [ ] 프로토타입용 시각 기준점으로 바닥 그림자, 높이 기준선, 또는 마커 주변 기준 오브젝트를 추가한다.

검증 기준:
- 오브젝트의 월드 좌표는 하나만 존재하고, 관객별 차이는 카메라 시점 차이에서 발생해야 한다.
- 오브젝트 등장, 이동, 회전, 사라짐이 공통 이벤트 타임라인에 맞춰 재생되어야 한다.
- 관객이 움직이거나 시선을 돌릴 때 오브젝트가 공연장 공간 안에 고정된 것처럼 보여야 한다.

### 기술 가정 및 검증 필요 항목
- 가정: 초기 프로토타입은 실제 공연장 전체가 아니라 축소된 테스트 공간 또는 Unity 시뮬레이션 씬에서 검증한다.
- 가정: 관객 눈 위치는 실제 안구 추적이 아니라 디바이스 카메라 위치와 눈 높이 오프셋으로 근사한다.
- 검증 필요: 중앙 마커 인식 거리, 조명 변화, 관객 밀집 상황에서의 인식 안정성.
- 검증 필요: 스마트 글래스 시야각 안에서 오브젝트 크기와 등장 위치가 과도하게 부담스럽지 않은지.
- AI에게 요청할 부분: 후속 단계에서 위 메커니즘을 Unity C# 컴포넌트 구조와 테스트 씬 구성안으로 분해한다.

## 기술 명세

### 작성 기준
- 본 기술 명세는 Unity 단일 씬 또는 에디터 내 다중 관객 시뮬레이션으로 핵심 경험을 검증하기 위한 프로토타입 기준이다.
- 기준 좌표계는 `Stage Space`이며, 모든 관객 위치, 눈 위치, 중앙 마커, 3D 오브젝트, 이벤트 상태는 먼저 `Stage Space` 값으로 정의한다.
- Unity 씬의 Transform은 `Stage Space`를 시각화하고 렌더링하기 위한 표현 계층으로 사용한다. 좌표 변환 지점은 코드와 Inspector 이름에 명확히 드러나야 한다.
- 모든 새 C# 클래스명과 파일명은 `SUN_` prefix를 사용한다.
- 실제 AR SDK, 네트워크 동기화, 디바이스별 캘리브레이션 정밀도는 후속 검증 대상으로 두고, 이번 명세에서는 대체 가능한 인터페이스와 수동 시뮬레이션 흐름을 우선한다.
- 개발자는 먼저 동일 오브젝트가 관객 위치에 따라 다른 시점으로 보이는지, 그 다음 중앙 마커 정렬과 이벤트 타임라인이 체험적으로 동기화되는지 검증한다.

#### 데이터 단위와 좌표 기준
- 거리 단위는 미터(m)를 사용한다. 변수명에는 필요한 경우 `Meters`를 붙인다.
- 시간 단위는 초(s)를 사용한다. 변수명에는 필요한 경우 `Seconds`를 붙인다.
- 회전은 내부 계산에서 `Quaternion`을 사용하고, Inspector 표시나 수동 입력에는 Euler degree 값을 허용한다.
- `Stage Space` 축 기준은 다음과 같이 정의한다.
  - 원점: 중앙 마커 중심 `(0, 0, 0)`.
  - `+Y`: 공연장 위쪽 방향.
  - `+Z`: 중앙 마커에서 무대 중심을 바라보는 방향.
  - `+X`: 중앙 마커에서 무대를 바라봤을 때 오른쪽 방향.
- 관객 눈 위치는 `관객 Stage Space 기준 위치 + 눈 높이 오프셋 + 디바이스 착용 오프셋`으로 계산한다.
- 3D 오브젝트의 권위 있는 위치와 회전은 항상 `Stage Space` 값이다. 관객별 차이는 오브젝트 데이터를 복제해서 만드는 것이 아니라 카메라 위치와 좌표 변환 결과로 만든다.

#### Unity 생명주기 사용 기준
- `Awake`: 같은 GameObject 또는 자식 오브젝트의 내부 참조를 캐시한다. 런타임에 반복 검색하지 않는다.
- `Start`: 다른 컴포넌트, 테스트 씬 컨트롤러, ScriptableObject 설정 등 외부 의존성이 필요한 초기화를 수행한다.
- `Update`: 수동 입력, 타임라인 진행 시간, 캘리브레이션 상태 변경 요청을 처리한다.
- `LateUpdate`: 관객 카메라 Transform, 렌더링 직전 오브젝트 Transform 반영, 디버그 표시 갱신처럼 프레임 최종 상태가 필요한 작업을 수행한다.
- 매 프레임 문자열 기반 검색, 불필요한 `FindObjectOfType`, 리스트 재할당, LINQ 기반 반복 할당은 피한다.

#### 검증 원칙
- 같은 `Stage Space` 오브젝트를 최소 2개 이상의 관객 위치에서 바라보는 장면을 기본 검증 단위로 삼는다.
- 검증은 `좌표계 정렬 -> 관객 눈 위치 계산 -> 동일 오브젝트 렌더링 -> 이벤트 동기화 -> 재정렬/리셋` 순서로 진행한다.
- 모든 검증 결과는 개발자가 씬 뷰 또는 게임 뷰에서 확인할 수 있어야 하며, 숫자 좌표와 시각 결과를 함께 본다.
- 관객별로 오브젝트 복사본을 따로 움직여서 시점 차이를 흉내 내지 않는다. 오브젝트의 공연장 기준 상태는 하나여야 한다.
- 정상 상태뿐 아니라 마커 미정렬, 시야 밖 오브젝트, 이벤트 일시정지, 리셋 후 재생 상황을 함께 확인한다.
- Unity 컴파일이 가능한 구조인지 확인하고, 실제 AR SDK가 없는 경우에도 에디터 시뮬레이션으로 핵심 로직이 실행되어야 한다.

#### 응답 원칙
- 필수 참조가 없거나 캘리브레이션이 완료되지 않은 경우, 컴포넌트는 조용히 실패하지 않고 디버그 오버레이에 상태를 표시한다.
- 좌표 또는 FOV 입력값이 허용 범위를 벗어나면 즉시 크래시하지 않고 프로토타입 기본값으로 보정하되, 보정 사실을 경고로 남긴다.
- 중앙 마커 추적이 끊기면 마지막 유효 정렬값을 유지하고, 개발자가 재스캔 또는 수동 리셋을 실행할 수 있게 한다.
- 이벤트 데이터가 누락되거나 대상 오브젝트 ID를 찾지 못하면 해당 이벤트만 건너뛰고, 타임라인 전체를 중단하지 않는다.
- 관객 카메라가 오브젝트를 시야 밖으로 벗어나게 하더라도 오브젝트를 화면에 억지로 붙이지 않는다. 공간 고정감을 우선한다.
- 개발자가 원인 파악을 할 수 있도록 주요 상태는 `Ready`, `Uncalibrated`, `Calibrating`, `TrackingLost`, `Playing`, `Paused`, `Reset` 중 하나로 표현한다.

#### 프로토타입 제외 범위
- 상용 수준의 멀티플레이 서버, 계정, 권한, 결제, 운영 자동화는 제외한다.
- 실제 공연장 전체 스캔, 고정밀 SLAM 맵 공유, 수백 명 이상 동시 접속 검증은 제외한다.
- 정밀 양안 렌더링, 안구 추적, 디바이스별 광학 보정은 제외하고 단일 대표 카메라로 검증한다.
- 무대 음향/조명 장비와의 실제 하드웨어 연동은 제외하고, 이벤트 타임라인의 논리적 동기화만 확인한다.

### 1. 공간 좌표계 기술 명세

#### 클래스 구조
- `SUN_StageCoordinateSystem`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `Transform _stageOrigin`, `Vector3 _stageForward`, `Vector3 _stageRight`, `float _metersToUnityScale`.
  - public API 의도: `StageToWorldPosition`, `WorldToStagePosition`, `StageToWorldRotation`, `WorldToStageRotation`.
  - 역할: 공연장 기준 좌표와 Unity 월드 좌표 사이의 단일 변환 진입점.
- `SUN_AudienceProfile`
  - 부모 클래스: `ScriptableObject`
  - Inspector 필드: `string _audienceId`, `Vector3 _seatStagePositionMeters`, `float _eyeHeightMeters`, `Vector3 _deviceOffsetMeters`, `float _fieldOfViewDegrees`.
  - 역할: 관객별 위치/시야 설정 프리셋.
- `SUN_AudienceRig`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `SUN_AudienceProfile _profile`, `Transform _cameraRoot`, `SUN_StageCoordinateSystem _coordinateSystem`.
  - 역할: 관객 프로필을 기반으로 눈 위치와 카메라 루트 위치를 계산.

#### Stage Space 기준 좌표 정의
- 중앙 마커 중심을 `(0, 0, 0)`으로 둔다.
- 무대 중심은 `+Z` 방향에 위치한다. 예: 중앙 마커에서 5m 앞이면 `(0, 0, 5)`.
- 객석 또는 스탠딩 관객은 중앙 마커 기준 뒤쪽이나 좌우에 배치될 수 있다. 예: 왼쪽 뒤 관객은 `(-2, 0, -4)`.
- 높이값은 `Y`축에만 반영한다. 관객 몸 기준 위치의 `Y`는 바닥 높이이며, 눈 위치 계산 시 눈 높이를 더한다.
- Unity 씬에서 실제 배율이 달라도 명세상 좌표값은 미터 기준으로 유지하고, `SUN_StageCoordinateSystem`에서 스케일 변환한다.

#### 관객 위치 및 눈 높이 오프셋
- 관객 몸 기준 위치: `SUN_AudienceProfile._seatStagePositionMeters`.
- 눈 높이: 기본값 1.6m를 사용하되 관객별로 조정 가능하게 둔다.
- 디바이스 착용 오프셋: 스마트 글래스 카메라가 실제 눈 중심보다 앞/위에 있다고 가정할 때 보정값으로 사용한다.
- 최종 눈 위치 계산식:
  - `eyeStagePositionMeters = seatStagePositionMeters + Vector3.up * eyeHeightMeters + deviceOffsetMeters`.
- 프로토타입에서는 관객 움직임을 직접 추적하지 않아도 되지만, 수동 슬라이더 또는 Transform 조작으로 관객 위치 변경을 테스트할 수 있어야 한다.

#### 좌표 변환 흐름
- 개발자는 모든 입력 데이터를 먼저 `Stage Space`로 저장한다.
- 렌더링 직전 `SUN_StageCoordinateSystem`을 통해 Unity 월드 Transform으로 변환한다.
- 관객 카메라는 변환된 눈 위치에 배치하고, 오브젝트는 변환된 `Stage Space` 위치에 배치한다.
- 디바이스 로컬 좌표계가 필요한 경우 `SUN_CentralMarkerCalibrator`의 보정 행렬을 거쳐 `Stage Space -> Device Local -> Unity World` 순서로 적용한다.
- 좌표 변환 함수는 단방향 계산을 흩어 쓰지 않고 `SUN_StageCoordinateSystem`에 모은다.

#### 검증 원칙
- 관객 A와 관객 B의 `seatStagePositionMeters`를 다르게 입력했을 때, 같은 오브젝트가 서로 다른 화면 위치와 원근감으로 보여야 한다.
- 관객 위치를 바꿔도 오브젝트의 `Stage Space` 좌표값은 변경되지 않아야 한다.
- `Stage Space` 원점, X/Y/Z 축, 무대 중심, 테스트 관객 위치를 Gizmo로 확인할 수 있어야 한다.
- Unity 월드 좌표를 직접 수정했을 때와 `Stage Space` 값을 수정했을 때의 차이를 개발자가 구분할 수 있어야 한다.

#### 응답 원칙
- `SUN_StageCoordinateSystem` 참조가 없으면 관객 카메라와 오브젝트 Presenter는 갱신을 중단하고 `Uncalibrated` 또는 `MissingCoordinateSystem` 상태를 표시한다.
- 스케일 값이 0 이하이면 1.0으로 대체하고 경고를 표시한다.
- 관객 프로필이 비어 있으면 기본 테스트 관객 위치 `(0, 0, -3)`과 눈 높이 1.6m를 사용하되, 디버그 오버레이에 기본값 사용 사실을 표시한다.

### 2. 중앙 마커 캘리브레이션 기술 명세

#### 클래스 구조
- `SUN_CentralMarkerCalibrator`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `Transform _markerVisual`, `SUN_StageCoordinateSystem _coordinateSystem`, `bool _useManualCalibration`, `Vector3 _manualMarkerPositionMeters`, `Vector3 _manualMarkerEulerDegrees`.
  - public API 의도: `BeginCalibration`, `ApplyMarkerPose`, `ResetCalibration`, `HasValidCalibration`.
  - 역할: 중앙 마커 인식값 또는 수동 입력값을 받아 디바이스 로컬 좌표계를 `Stage Space`에 맞춘다.
- `SUN_PrototypeDebugOverlay`
  - 부모 클래스: `MonoBehaviour`
  - 역할: 마커 인식 여부, 보정 행렬, 재스캔 필요 상태를 표시한다.

#### 마커 기준 위치와 방향 정의
- 마커 중심은 `Stage Space` 원점이다.
- 마커의 전방 방향은 무대 중심을 바라보는 `+Z` 축과 일치해야 한다.
- 마커 시각 오브젝트에는 방향을 알 수 있는 표식을 둔다. 예: 무대 방향 화살표, X/Z 축 색상 표시.
- 실제 AR 마커 인식이 없을 경우, `_manualMarkerPositionMeters`와 `_manualMarkerEulerDegrees`로 동일 흐름을 시뮬레이션한다.

#### 디바이스 로컬 좌표계 정렬 흐름
- 체험 시작 전 상태는 `Uncalibrated`다.
- 관객 디바이스가 중앙 마커를 바라보면 `BeginCalibration`을 호출한다.
- 마커 Pose를 얻으면 `ApplyMarkerPose`가 디바이스 로컬 원점과 `Stage Space` 원점 사이의 위치/회전 오프셋을 계산한다.
- 보정 성공 후 관객 눈 위치와 오브젝트 위치는 같은 보정값을 사용해 렌더링된다.
- 보정값은 관객별로 다를 수 있지만, 보정 후 참조하는 `Stage Space` 데이터는 동일해야 한다.

#### 재스캔 및 수동 리셋 흐름
- 재스캔은 기존 보정값을 즉시 삭제하지 않고, 새 마커 Pose가 유효할 때 교체한다.
- 수동 리셋은 상태를 `Uncalibrated`로 돌리고 관객 카메라와 오브젝트 표시를 검증용 기본 위치로 되돌린다.
- 추적 손실 상태에서는 마지막 유효 보정값을 유지하되 `TrackingLost` 경고를 표시한다.
- 재스캔 성공 시 이벤트 타임라인은 유지하고, 좌표계 정렬값만 갱신한다.

#### 검증 원칙
- 마커 정렬 전에는 오브젝트를 최종 위치에 표시하지 않거나 디버그용 미정렬 상태로 표시한다.
- 마커 정렬 후 여러 관객 카메라가 같은 `Stage Space` 원점을 공유해야 한다.
- 재스캔 후에도 오브젝트의 공연장 기준 위치와 이벤트 진행 상태가 바뀌지 않아야 한다.
- 마커 방향을 일부러 틀어 입력했을 때 오브젝트 방향도 함께 틀어지는지 확인해 보정 영향 범위를 검증한다.

#### 응답 원칙
- 마커 인식 실패 시 자동으로 임의 보정값을 만들지 않는다. 수동 리셋 또는 재스캔 요청 상태를 표시한다.
- 보정값이 없으면 관객별 렌더링을 `Uncalibrated` 상태로 두고, 이벤트 타임라인은 시작 가능하더라도 최종 AR 표시를 제한한다.
- 마커 Pose가 비정상적으로 큰 이동 또는 회전을 만들면 적용 전 경고를 표시하고 수동 승인 후 적용할 수 있게 한다.

### 3. 관객 카메라 시점 기술 명세

#### 클래스 구조
- `SUN_AudienceViewController`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `Camera _audienceCamera`, `SUN_AudienceRig _audienceRig`, `SUN_CentralMarkerCalibrator _calibrator`, `float _minFovDegrees`, `float _maxFovDegrees`.
  - public API 의도: `SetLookRotation`, `SetSimulatedHeadYawPitch`, `ApplyAudienceView`.
  - 역할: 관객 눈 위치와 시선 방향을 기반으로 최종 카메라 상태를 갱신한다.
- `SUN_AudienceRig`
  - 부모 클래스: `MonoBehaviour`
  - 역할: 관객의 `Stage Space` 눈 위치를 계산해 View Controller에 제공한다.

#### 관객 눈 위치 계산
- `SUN_AudienceRig`는 프로필의 좌석 위치, 눈 높이, 디바이스 착용 오프셋을 합산한다.
- 계산 결과는 `Stage Space` 기준 `Vector3`로 유지한다.
- `SUN_AudienceViewController`는 이 값을 `SUN_StageCoordinateSystem`과 `SUN_CentralMarkerCalibrator`를 통해 Unity 월드 카메라 위치로 변환한다.
- 관객 위치 수동 이동이 있을 경우, 다음 프레임 `LateUpdate`에서 카메라 위치가 반영되어야 한다.

#### 카메라 회전 및 시선 방향 갱신
- 프로토타입 입력은 마우스/키보드, Inspector 값, 또는 테스트 Transform 회전으로 대체할 수 있다.
- 시선 방향은 관객 로컬 회전으로 저장하되, 최종 렌더링 전 `Stage Space` 기준 회전으로 해석한다.
- 카메라가 오브젝트를 바라보도록 강제 LookAt을 걸지 않는다. 관객의 시선 입력과 오브젝트의 공간 고정감을 분리한다.
- 회전 업데이트는 사용자 입력 처리와 카메라 Transform 반영을 분리해 디버깅 가능하게 만든다.

#### 스마트 글래스 FOV 기준
- 기본 FOV는 프로토타입 가정값으로 45도에서 시작한다.
- 최소/최대 FOV는 Inspector에서 제한한다. 예: 30도 이상 70도 이하.
- 오브젝트가 시야 밖으로 나가는 상황은 오류가 아니라 검증 대상이다.
- FOV가 좁을 때 오브젝트 등장 위치가 관객에게 과도하게 부담스럽지 않은지 확인한다.

#### 검증 원칙
- 같은 `Stage Space` 오브젝트가 관객 A, B 카메라에서 서로 다른 각도와 거리감으로 보여야 한다.
- 관객 카메라를 회전해도 오브젝트가 화면 중심에 강제로 붙지 않아야 한다.
- FOV 값을 바꿨을 때 시야 내/외 판정과 체감 크기가 달라져야 한다.
- 눈 높이를 변경하면 오브젝트를 바라보는 수직 각도가 달라져야 한다.

#### 응답 원칙
- 카메라 참조가 없으면 해당 관객 View Controller만 비활성화하고 다른 관객 시뮬레이션은 계속 실행한다.
- FOV 입력값이 범위를 벗어나면 제한값으로 clamp하고 경고를 표시한다.
- 캘리브레이션이 유효하지 않으면 마지막 유효 카메라 위치를 유지하거나 테스트 기본 위치로 전환하며, 상태를 명확히 표시한다.

### 4. 이벤트 동기화 기술 명세

#### 클래스 구조
- `SUN_StageEventDefinition`
  - 부모 클래스: 없음, `[System.Serializable]` 데이터 클래스
  - 필드: `string EventId`, `string ObjectId`, `float StartTimeSeconds`, `float DurationSeconds`, `Vector3 StartPositionMeters`, `Vector3 EndPositionMeters`, `Vector3 StartEulerDegrees`, `Vector3 EndEulerDegrees`, `Vector3 StartScale`, `Vector3 EndScale`, `bool IsVisible`.
  - 역할: 하나의 연출 이벤트를 데이터로 정의한다.
- `SUN_StageObjectState`
  - 부모 클래스: 없음, `[System.Serializable]` 데이터 클래스
  - 필드: `string ObjectId`, `bool IsVisible`, `Vector3 StagePositionMeters`, `Quaternion StageRotation`, `Vector3 Scale`, `float NormalizedProgress`.
  - 역할: 특정 프레임에 렌더러가 적용할 최종 상태값을 담는다.
- `SUN_StageEventTimeline`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `List<SUN_StageEventDefinition> _events`, `bool _playOnStart`, `float _timelineTimeSeconds`.
  - public API 의도: `Play`, `Pause`, `ResetTimeline`, `EvaluateObjectState`.
  - 역할: 공통 이벤트 시간을 관리하고 오브젝트별 상태값을 평가한다.

#### 이벤트 데이터 구조
- 이벤트는 오브젝트 ID를 기준으로 대상 Presenter에 연결된다.
- 위치와 회전은 `Stage Space` 기준으로 정의한다.
- 진행률은 `StartTimeSeconds`와 `DurationSeconds`로 계산한다.
- 프로토타입에서는 선형 보간을 기본으로 사용하고, 복잡한 커브는 후속 작업으로 분리한다.
- 이벤트 데이터는 관객별로 복제하지 않는다. 모든 관객은 같은 이벤트 정의와 같은 진행 시간을 참조한다.

#### 공통 타임라인 제어
- `Play`: 현재 시간부터 이벤트 평가를 시작한다.
- `Pause`: 시간 증가를 멈추고 현재 상태값을 유지한다.
- `ResetTimeline`: 진행 시간을 0으로 되돌리고 초기 상태를 재적용한다.
- `EvaluateObjectState`: 특정 ObjectId와 현재 시간을 입력받아 해당 오브젝트의 표시 상태, 위치, 회전, 크기를 반환한다.
- 네트워크 동기화 대신 에디터 또는 로컬 씬에서 하나의 타임라인 컴포넌트를 모든 관객이 공유한다.

#### 관객별 렌더링 상태 참조 흐름
- `SUN_StageEventTimeline`이 공통 상태값을 만든다.
- `SUN_ARObjectPresenter`가 해당 상태값을 읽어 오브젝트 Transform에 적용한다.
- 관객별 카메라는 같은 오브젝트 Transform을 각자의 위치에서 바라본다.
- 관객마다 별도의 오브젝트 상태를 계산해야 하는 경우에도, 그 차이는 렌더링 보정이나 디버그 표시로 한정하고 원본 이벤트 상태는 바꾸지 않는다.

#### 검증 원칙
- 관객 위치가 달라도 이벤트 시작, 이동, 회전, 사라짐 타이밍은 동일해야 한다.
- 타임라인을 일시정지하면 모든 관객 시점에서 오브젝트 상태가 같은 순간에 멈춰야 한다.
- 리셋 후 다시 재생했을 때 같은 이벤트가 같은 결과로 반복되어야 한다.
- 이벤트 ID, ObjectId, 현재 진행 시간을 디버그 오버레이에서 확인할 수 있어야 한다.

#### 응답 원칙
- 이벤트 지속 시간이 0 이하이면 즉시 완료 이벤트로 처리하고 경고를 표시한다.
- 대상 ObjectId가 없는 이벤트는 건너뛰고, 누락된 ID를 디버그 오버레이에 표시한다.
- 타임라인이 `Paused` 상태일 때 수동으로 관객 위치를 바꾸면, 이벤트 상태는 유지하고 관객 시점 변화만 반영한다.
- 중복 EventId가 있으면 실행은 가능하게 두되 경고를 표시하고 후속 데이터 정리를 요청한다.

### 5. 3D 오브젝트 렌더링 기술 명세

#### 클래스 구조
- `SUN_ARObjectPresenter`
  - 부모 클래스: `MonoBehaviour`
  - Inspector 필드: `string _objectId`, `Transform _targetObject`, `Renderer[] _renderers`, `SUN_StageCoordinateSystem _coordinateSystem`, `SUN_StageEventTimeline _timeline`.
  - public API 의도: `ApplyStageObjectState`, `SetVisible`, `SetStageTransform`.
  - 역할: 공통 이벤트 상태를 실제 3D 오브젝트 렌더링에 적용한다.
- `SUN_PrototypeSceneController`
  - 부모 클래스: `MonoBehaviour`
  - 역할: 테스트용 오브젝트 프리셋, 시작 위치, 이벤트 연결 상태를 관리한다.

#### 오브젝트 Stage Space 배치 기준
- 테스트 오브젝트는 최소 1개 이상 배치한다. 예: 중앙 마커 위 2m 지점 `(0, 2, 0)` 또는 무대 앞 공중 `(0, 3, 4)`.
- 오브젝트의 원본 위치, 회전, 크기는 `Stage Space` 기준 데이터로 기록한다.
- Unity Transform 값은 이벤트 상태 적용 결과일 뿐, 원본 데이터로 사용하지 않는다.
- 오브젝트 ID는 이벤트 데이터의 `ObjectId`와 정확히 일치해야 한다.

#### 이벤트 상태 기반 위치, 회전, 크기 갱신
- 매 프레임 `SUN_StageEventTimeline`에서 오브젝트 ID에 해당하는 `SUN_StageObjectState`를 평가한다.
- Presenter는 상태값의 위치/회전/크기를 `SUN_StageCoordinateSystem`으로 변환해 `_targetObject`에 적용한다.
- 표시 상태가 false이면 Renderer를 비활성화하되, Transform 정보는 유지한다.
- 그림자, 기준선, 보조 마커는 오브젝트의 공간감을 검증하기 위한 별도 시각 요소로 관리한다.

#### 관객별 원근 렌더링 기준
- 오브젝트는 카메라를 향해 자동 회전하는 빌보드가 아니다.
- 관객별 차이는 카메라 위치와 회전에서 발생해야 한다.
- 오브젝트가 관객에게 너무 가깝거나 시야 밖에 있으면 그 상태를 그대로 유지하고 디버그 표시로만 안내한다.
- 필요 시 테스트용으로만 오브젝트 바운딩 박스, 거리 라벨, Stage Space 좌표 라벨을 표시한다.

#### 시각 기준점과 공간감 보조 요소
- 중앙 마커 주변에 바닥 그리드 또는 축 표시를 둔다.
- 오브젝트 아래에는 선택적으로 그림자 또는 높이 기준선을 둔다.
- 무대 중심, 관객 위치, 오브젝트 위치 사이의 관계를 씬 뷰 Gizmo로 확인할 수 있게 한다.
- 보조 요소는 최종 연출물이 아니라 검증 도구이므로 런타임 토글로 켜고 끌 수 있어야 한다.

#### 검증 원칙
- 오브젝트의 월드 좌표가 아니라 `Stage Space` 좌표가 권위 있는 원본인지 확인한다.
- 같은 오브젝트를 관객별로 복제해 각도 차이를 만들지 않는다.
- 오브젝트 등장, 이동, 회전, 사라짐이 공통 타임라인과 일치해야 한다.
- 관객이 이동하거나 시선을 돌려도 오브젝트가 공연장 공간에 고정된 대상으로 느껴져야 한다.

#### 응답 원칙
- `_targetObject`가 없으면 해당 Presenter는 비활성화하고 ObjectId와 오류 상태를 표시한다.
- Renderer 참조가 비어 있으면 Transform 갱신은 계속하되 표시 상태 제어 경고를 남긴다.
- 이벤트 상태가 없으면 마지막 유효 상태를 유지하거나 초기 상태로 되돌리고, 어떤 정책을 사용했는지 오버레이에 표시한다.

### 6. 프로토타입 테스트 씬 기술 명세

#### 테스트 씬 구성 항목
- 중앙 마커 원점 오브젝트 1개.
- 무대 중심 기준 오브젝트 1개.
- 테스트 관객 Rig 최소 2개. 예: 왼쪽 뒤 관객 A, 오른쪽 뒤 관객 B.
- 관객별 카메라 2개 또는 화면 분할 뷰.
- 테스트용 3D 오브젝트 1개 이상.
- 공통 이벤트 타임라인 1개.
- 디버그 오버레이와 수동 제어 UI.

#### 디버그 표시 항목
- `Stage Space` 원점과 X/Y/Z 축.
- 관객별 몸 위치, 눈 위치, 시선 방향.
- 중앙 마커 캘리브레이션 상태와 마지막 유효 보정 시간.
- 이벤트 ID, ObjectId, 타임라인 진행 시간, 진행률.
- 오브젝트 Stage Space 좌표, Unity 월드 좌표, 관객과의 거리.
- 경고 상태: 미정렬, 추적 손실, 누락된 참조, 시야 밖, 이벤트 데이터 오류.

#### 수동 제어 항목
- 타임라인 `Play`, `Pause`, `Reset`.
- 중앙 마커 `Calibrate`, `Rescan`, `Manual Reset`.
- 관객 A/B 위치 이동.
- 관객 A/B 시선 yaw/pitch 조정.
- FOV 조정.
- 디버그 표시 On/Off.

#### 검증 원칙
- 한 씬에서 좌표계, 관객 시점, 오브젝트 렌더링, 이벤트 동기화를 모두 확인할 수 있어야 한다.
- 테스트는 관객 A와 B의 화면 결과를 나란히 비교할 수 있어야 한다.
- 수동 리셋 후 같은 입력값으로 같은 결과가 반복되어야 한다.
- 디버그 표시를 꺼도 핵심 연출 흐름은 정상 동작해야 한다.

#### 응답 원칙
- 테스트 씬 시작 시 필수 참조를 검사하고 누락 항목을 목록으로 표시한다.
- 일부 관객 Rig가 실패해도 나머지 관객 Rig와 타임라인 검증은 계속 진행한다.
- 수동 제어 입력은 현재 상태에 맞지 않더라도 크래시하지 않고 가능한 상태 전환만 수행한다. 예: 미정렬 상태에서 재생 버튼을 누르면 타임라인은 재생하되 AR 표시 상태는 미정렬 경고를 유지한다.

### 7. 기술 가정 및 검증 필요 항목

#### 기술 가정
- 초기 검증은 Unity 에디터 또는 단일 PC에서 여러 관객 카메라를 동시에 시뮬레이션한다.
- 실제 스마트 글래스의 추적값은 `SUN_CentralMarkerCalibrator`와 `SUN_AudienceViewController`에 연결 가능한 외부 입력으로 나중에 대체한다.
- 관객 눈 위치는 실제 안구 중심이 아니라 디바이스 착용 위치와 눈 높이 오프셋으로 근사한다.
- 이벤트 동기화는 로컬 공통 타임라인 기준으로 검증하고, 네트워크 시간 동기화는 후속 단계에서 다룬다.
- 오브젝트 렌더링 품질보다 위치, 각도, 타이밍의 체험 검증을 우선한다.

#### 검증 필요 항목
- 중앙 마커가 실제 공연장 조명, 거리, 관객 밀집 환경에서 안정적으로 인식되는지.
- 관객별 캘리브레이션 오차가 동일 오브젝트의 공간 고정감에 어느 정도 영향을 주는지.
- 스마트 글래스 FOV가 좁을 때 오브젝트 등장 위치와 이동 속도가 부담스럽지 않은지.
- 여러 관객이 동시에 같은 이벤트를 경험할 때 허용 가능한 시간 오차 범위가 어느 정도인지.
- 프로토타입에서 사용하는 단일 카메라 근사가 실제 양안 AR 디바이스 체감과 얼마나 다른지.

#### AI에게 요청할 항목
- 위 클래스 구조를 기준으로 Unity C# 스크립트 초안을 작성한다.
- 테스트 씬 Hierarchy 구성안과 프리팹 구조를 제안한다.
- `SUN_StageCoordinateSystem`의 좌표 변환 함수와 단위 테스트 예시를 작성한다.
- 관객 A/B/C 시뮬레이션용 샘플 `SUN_AudienceProfile` 값을 생성한다.
- 이벤트 타임라인 샘플 데이터와 디버그 오버레이 표시 문구를 작성한다.
